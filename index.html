Modules 6–7
What is IAM?
DefinitionEnsures the right person gets the right access to the right resource at the right time.

Core components (4As)

IdentityWho you are — user, system, or service.
AuthenticationProving your identity.
AuthorizationWhat you are allowed to do.
Accounting/AuditingTracking what you did.
Access Controls (10 Types)
Preventative — stops incidents before they happen
Detective — detects incidents when they occur
Corrective — corrects after an incident
Deterrent — discourages bad behavior
Recovery — restores after an incident
Directive — directs subjects toward compliance
Compensating — compensates for a weakness
Administrative — policies, procedures, training
Technical/Logical — software & hardware controls
Physical — physical barriers and controls
Access Models
RuBACRule-Based Access Control — access decided by predefined rules/conditions. Supports dynamic & fine-grained control.
ABACAttribute-Based Access Control — decisions based on attributes of users, resources, and environment. More flexible than roles alone.
Risk-Based ACDynamically adjusts permissions based on risk level of the access request (sensitivity, behavior, security posture).
PEPPolicy Enforcement Point — intercepts requests and enforces policies.
PDPPolicy Decision Point — evaluates policies and makes allow/deny decisions.
PIPPolicy Information Point — provides attribute info to PDPs.
PAPPolicy Administration Point — where admins define and manage access policies.
Provisioning & Account Lifecycle
ProvisioningOnboarding — adding a user to the IAM system and granting access.
DeprovisioningOffboarding — removing a user: disable/delete account, revoke certs, cancel access codes, return equipment.
Privilege CreepWhen users accumulate access rights over time beyond what they need. Account management/reviews help guard against this.
Account Access ReviewVerifies accounts don't have excessive privileges; ensures policies are followed; confirms accounts are disabled/deleted as required.
Exam tip: Deprovisioning issues arise when employees are "tipped off" early — e.g., IT requests equipment back prematurely, which is a security risk.
Service Accounts
DefinitionAlso called application accounts. Digital identity used by software/services to interact with other apps or the OS. Often used for machine-to-machine (API) communication.

Attack vectors on service accounts

Privilege EscalationAttacker impersonates the account to elevate privileges.
SpoofingImpersonates the account to hide identity.
Non-repudiationUses account to hide actions performed on the system.
Information DisclosureExtracts infrastructure information via the service account.
Session Management
Session CreationSession IDs identify the session.
Session TrackingSession ID tracks relevant user data.
Session TimeoutTerminates inactive sessions automatically.
Session TerminationUser manually logs out.
Session SecurityProtects against hijacking and fixation attacks.
Client-sideSession data stored/managed on the client.
Server-sideSession data stored/managed on the server.
FIM — Federated Identity Management
IdPIdentity Provider — authenticates users and issues security tokens.
SPService Provider — hosts services; relies on IdP tokens to grant access.
Security TokenIssued by IdP after authentication; contains identity and permissions (SAML or JWT format).
Federation ProtocolStandardized comms between IdP and SP — OpenID Connect, OAuth.

FIM deployment options

On-premiseAll FIM software on-site.
CloudThird-party FIM in the cloud.
HybridMix of on-premise and cloud.

Benefits of FIM

OAuth 2.0, OpenID, OIDC
OAuth 2.0Open Authorization. RFC 6749, maintained by IETF. Used to share account info with 3rd party sites. NOT backward compatible with OAuth 1.0.
OpenIDOpen standard by OpenID Foundation. Decentralized authentication — one set of credentials for multiple sites via an OpenID Provider.
OIDCOpenID Connect — authentication layer ON TOP of OAuth 2.0. Both auth & authorization. Uses JWT. Example: logging into eBay with your Google account.
Exam tip: OAuth = authorization only. OpenID = authentication only. OIDC = both.
SAML
Full NameSecurity Assertion Markup Language — based on XML. Used to exchange auth/authorization info between federated orgs. Often used for browser-based SSO.

SAML 2.0 entities

Principal / User AgentThe user trying to access an account.
Service Provider (SP)The website the user is accessing.
Identity Provider (IdP)Third party holding the user's auth and authorization info.
Kerberos
TypeSSO system used within networks. Ticket authentication mechanism using a trusted third party.
KDCKey Distribution Center — trusted third party; all clients/servers register here.
Auth ServerHosts KDC functions; verifies authenticity and timeliness of tickets.
TicketEncrypted message proving subject is permitted to access an object.
TGTTicket-Granting Ticket — proof that subject authenticated via KDC and can request access to other objects.
Kerberos RealmA logical area controlled by Kerberos.

RADIUS
TACACS+
Remote Authentication Dial-in User Service
Terminal Access Controller Access Control System Plus (Cisco)
Uses UDP

Security Testing vs Assessment vs Audit
Security Testing
Security Assessment
Verifies controls are functioning properly
Comprehensive review — goes beyond testing
Should be on a regular schedule
Includes risk and threat environment reviews
Focused on specific controls
Assessment report often required
Security AuditSimilar to assessment but performed by independent auditors. Can be internal, external, or third-party.
Key distinction: Testing = verify controls. Assessment = comprehensive + risk. Audit = assessment done by independent auditors.

Penetration Testing Types
Type
Description
Black Box
Tester has zero knowledge of the system — simulates an external attacker.
White Box
Tester has full knowledge — source code, architecture, credentials. Most thorough.
Gray Box
Tester has partial knowledge — simulates an insider or partially informed attacker.
Memory trick: Black = zero info (no light). White = full info (full light). Gray = somewhere in between.

The main goal of test data analysis = a REMEDIATION ROADMAP provided to stakeholders.
Exception Handling & Ethical Disclosure
ExceptionWhen a policy, procedure, or control is temporarily bypassed. Must be documented and justified. Exceptions always increase risk.
Ethical DisclosureSome security incidents must be ethically disclosed. Many orgs lack a disclosure policy. DHS created a Vulnerability Disclosure Policy template.
Exam tip: Exceptions = temporary bypass = increased risk. Documentation is always key.
