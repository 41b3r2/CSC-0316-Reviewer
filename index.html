What is IAM?
DefinitionEnsures the right person gets the right access to the right resource at the right time.

Core components (4As)

IdentityWho you are — user, system, or service.
AuthenticationProving your identity.
AuthorizationWhat you are allowed to do.
Accounting/AuditingTracking what you did.
Assets to Protect (Physical & Logical Access)
Information — all data of the organization
Systems — all hardware of IT
Devices — compute nodes including BYOD
Facilities — physical locations owned or rented
Applications — programs performing specific functions
Services — software components providing specific functions
Access Controls (10 Types)
Preventative — stops incidents before they happen
Detective — detects incidents when they occur
Corrective — corrects after an incident
Deterrent — discourages bad behavior
Recovery — restores after an incident
Directive — directs subjects toward compliance
Compensating — compensates for a weakness
Administrative — policies, procedures, training
Technical/Logical — software & hardware controls
Physical — physical barriers and controls
Access Models
RuBACRule-Based Access Control — access decided by predefined rules/conditions. Supports dynamic & fine-grained control.
ABACAttribute-Based Access Control — decisions based on attributes of users, resources, and environment.
Risk-Based ACDynamically adjusts permissions based on risk level of the access request.
PEPPolicy Enforcement Point — intercepts requests and enforces policies.
PDPPolicy Decision Point — evaluates policies and makes allow/deny decisions.
PIPPolicy Information Point — provides attribute info to PDPs.
PAPPolicy Administration Point — where admins define and manage access policies.
Provisioning & Deprovisioning
ProvisioningOnboarding — adding a user and granting access.
DeprovisioningOffboarding — disable/delete account, revoke certs, cancel access codes, return equipment.
Privilege CreepWhen users accumulate access rights beyond what they need over time.
Security risk: employees "tipped off" before deprovisioning (premature equipment return, disabled accounts, revoked badges, leaked reorganization chart).
Service Accounts
DefinitionAlso called application accounts. Digital identity used by software/services for app-to-app or API communication.

Attack vectors

Privilege Escalation — impersonate account to elevate privileges
Spoofing — impersonate account to hide identity
Non-repudiation — hide actions using the service account
Information Disclosure — extract infrastructure info via service account
Session Management
Session CreationSession IDs identify the session.
Session TrackingSession ID tracks relevant user data.
Session TimeoutTerminates inactive sessions automatically.
Session TerminationUser manually logs out.
Session SecurityProtects against hijacking and fixation attacks.
Client-sideSession data stored/managed on the client.
Server-sideSession data stored/managed on the server.
Federated Identity Management (FIM)
IdPIdentity Provider — authenticates users and issues security tokens.
SPService Provider — hosts services; relies on IdP tokens to grant access.
Security TokenIssued by IdP after authentication; contains identity and permissions (SAML or JWT).
Federation ProtocolStandardized comms between IdP and SP — OpenID Connect, OAuth.

FIM deployment options

On-premiseAll FIM software on-site.
CloudThird-party FIM in the cloud.
HybridMix of on-premise and cloud.

Benefits of FIM

SSO — authenticate once, access multiple services
Improved Security — centralized auth/authorization
Interoperability — seamless integration across domains
Reduced Admin Overhead — streamlined provisioning/deprovisioning
SSO vs JIT
SSO
JIT
Authenticate once → access multiple apps
Temporary access granted only when needed
Reduces password fatigue

Risk: single point of failure

OAuth 2.0 / OpenID / OIDC
OAuth 2.0Open Authorization. RFC 6749, maintained by IETF. Shares account info with 3rd party sites. NOT backward compatible with OAuth 1.0.
OpenIDOpen standard by OpenID Foundation. Decentralized authentication — one set of credentials for multiple sites.
OIDCOpenID Connect — authentication layer ON TOP of OAuth 2.0. Both auth & authorization. Uses JWT. Example: eBay with Google account.
OAuth = authorization only. OpenID = authentication only. OIDC = both.
SAML & Kerberos
SAMLSecurity Assertion Markup Language — based on XML. Exchanges auth/authorization info between federated orgs. Used for browser-based SSO.

SAML 2.0 entities

Principal/User AgentThe user trying to access an account.
Service Provider (SP)The website the user is accessing.
Identity Provider (IdP)Third party holding the user's auth and authorization info.

Kerberos

KDCKey Distribution Center — trusted third party; all clients/servers register here.
Auth ServerHosts KDC functions; verifies authenticity and timeliness of tickets.
TicketEncrypted message proving subject is permitted to access an object.
TGTTicket-Granting Ticket — proof that subject authenticated via KDC.
Kerberos RealmA logical area controlled by Kerberos.
RADIUS vs TACACS+
RADIUS
TACACS+
Remote Authentication Dial-in User Service
Terminal Access Controller Access Control System Plus (Cisco)
Uses UDP
Uses TCP port 49
Only encrypts password exchange
Encrypts ALL authentication info by default

TACACS+ = more secure (full encryption), more flexible (separate AAA). RADIUS = simpler, UDP, password-only encryption.
Security Testing vs Assessment vs Audit
Security Testing
Security Assessment
Verifies controls are functioning properly
Comprehensive review — goes beyond testing
Should be on a regular schedule
Includes risk and threat environment reviews
Focused on specific controls
Assessment report often required
Security AuditSimilar to assessment but performed by independent auditors.
Testing = verify controls. Assessment = comprehensive + risk. Audit = assessment by independent auditors.
